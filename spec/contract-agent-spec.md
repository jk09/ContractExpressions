# Contract Agent Specification

## Overview

The Contract Agent system is an AI agent which adds `Design-by-Contract` functionality (as specified, e.g. by the Eiffel language) into a .NET program and writes tests which verify if the contracts are satisfied.

## Objectives
- Provides a Design-by-Contract framework to be consumed by programs
- Use the AI agent to decorate classes and methods with contracts (preconditions, postconditions, invariants)
- Use the AI agent to write unit tests which calls methods with generated arguments and verify if the contracts are satisfied.

## Functional Requirements

### Requirement FR-1: [Feature Name]

**Description:** The core of the system is an AI agent which generates a program based on the input specification. The specification are also used to infer contracts on the generated classes and methods, using the API of a separate `Design-by-Contract` framework which is part of the system.
After the program is generated, a test project is also generated. The test project contains unit tests for all classes and methods decorated with contracts. The unit tests are parameterized with input arguments for the methods (including constructor classes). 
The input arguments are reasonably exhaustive, inferred from the program specification, or randomly generated by a program like `FsCheck`.

The `Design-by-Contract` framework is specified in the [README](../../README.md). Its modification for the purpose of this system implementation is allowed. The baseline implementation is the project [ContractExpressions4](../../ContractExpressions4/ContractExpressions4.csproj)

**Input:**
- Parameter 1: the specification of a program, possibly including explicit contract specifications

**Output:**
- Result 1: the program matching the input specification
- Result 2: the contracts on classes and methods of the output program, based on specification, including explicit contracts specifications. The contracts are specified by the `Design-by-Contract` framework
- Result 3: an `xUnit` test project with parameterized unit tests. Each test verifies if a method contract are satisfied, by running by calling the method with some arguments and handle two scenarios:
        - if the arguments satisfy the preconditions and the target object satisfy invariants, the the method call must satisfy postconditions and the object invariant must be satisfied
        - if the preconditions are not satisfied, the contract failure exception is thrown

**Constraints:** N/A

**Examples:**

Input: `>> /contract-agent Write a program for division of two integers`
Output: 
    - a program such as 
    ```cs
    // Program.cs



    public class DivisionProgram : IDivisionProgram
    {
    
        public int Divide(int x, int y) => x / y;
    }

    // Program.Contracts.cs
    #define CONTRACTS_FULL

    using System.Diagnostics.Contracts;
    using ContractExpressions4; // the Design-by-Contract framework ContractExpressions4

    [ContractClass(typeof(DivisionProgramContracts))]
    public interface IDivisionProgram
    {
        int Divide(int x, int y);
    }

    [ContractClassFor(typeof(IDivisionProgram))]
    public class DivisionProgramContracts
    {
        public DivisionProgramContracts()
        {
            Dbc.Def(static (IDivisionProgram t, int x, int y) => t.Divide(x, y), 
                    static (IDivisionProgram t, int x, int y) => Contract.Requires(y != 0),
                    static (IDivisionProgram t, int x, int y) => Contract.Ensures(!(Math.Abs(x) < Math.Abs(y)) || Contract.Result<int>() == 0));
        }
    }

    // Program.Test.cs - own project
    
    using xUnit;
    using ContractExpressions4; // the Design-by-Contract framework ContractExpressions4


    public class DivisionProgramTests
    {
        [Theory]
        [InlineData(1, 2)]
        public void TestDivison_ContractSatsified(int x, int y)
        {
            var proxy = Dbc.Make<IDivisionProgram>(new DivisionProgram());
            proxy.Divide(x, y);
        }

        [Theory]
        [InlineData(1, 0)]
        public void TestDivision_ContractFailed(int x, int y)
        {
            var proxy = Dbc.Make<IDivisionProgram>(new DivisionProgram());
            ContractViolationException ex = Assert.Throws<ContractViolationException>(() => proxy.Divide(x, y));
            Assert.Equal(ContractKind.Precondition, ex.Kind);
        }
    }
    ```


## Non-Functional Requirements


### Reliability
- Availability: the contracts can be defined on any method and class, subject to being inferable from the specifications
- Error handling: handle the contract exceptions as defined in the `Design-by-Contract` framework.

### Security
- Authentication: N/A
- Data protection: the contracts are pure functions (predicates), which cannot modify state on which they act.



## API/Interface Specification

Use the `Dbc.Def(...)` API as defined by [README](../../README.md).
